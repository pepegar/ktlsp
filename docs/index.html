ktlsp

Fast Kotlin language tooling in Rust.

GitHub README

Toward a production-ready Kotlin LSP

Useful editor answers without dragging a full compiler stack into every request.

ktlsp is a lightweight Kotlin language server built around tree-sitter, a Rust core, and a conservative semantic model. It favors fast startup, fast navigation, and returning no result over returning the wrong result.

Why it exists

Keep the editing loop fast.

ktlsp is aimed at Kotlin developers who want responsive editor feedback and real source navigation without accepting heavyweight startup or opaque behavior.

What it does today

Cover the features that make daily navigation workable.

Goto definition, references, completion, hover, rename, semantic tokens, hierarchy features, import assistance, and conservative diagnostics are already in place.

What makes it credible

Validation goes through a real editor harness.

Editor-visible changes are meant to be proven through `dev/ktlsp-harness.sh`, not only through pure Rust tests, so regressions show up where users actually feel them.

How it ships

Release tags now produce downloadable binaries.

The initial release pipeline packages `ktlsp` for Linux, macOS, and Windows and publishes those archives to GitHub Releases with checksums.

Feature surface

Built around the practical LSP loop.

  • Workspace and library goto definition
  • Find references and rename
  • Completion and import actions
  • Hover, symbols, highlights
  • Semantic tokens and inlay hints
  • Selection and folding ranges
  • Implementation and type definition
  • Call and type hierarchy

Readiness

Not 1.0 yet, but pointed in the right direction.

The project should be presented honestly: strong performance characteristics, a growing editor-validation story, and clear gaps around semantic depth, compatibility guarantees, packaging, and release hardening.

Path forward

How ktlsp becomes production ready.

  1. Raise semantic correctness and tighten diagnostics behavior.
  2. Expand harness coverage for every editor-visible feature.
  3. Improve installation, packaging, and release discipline.
  4. Document compatibility, troubleshooting, and operational expectations.

Get involved

Use it, break it, validate it, and tighten it.

The shortest path to a stronger release is disciplined real-editor testing paired with clear documentation and focused fixes.
